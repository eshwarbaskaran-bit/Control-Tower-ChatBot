================================================================================
SECTION 1: GENERAL OVERVIEW
================================================================================

The ClickPost Control Tower is a real-time monitoring dashboard for logistics and operations teams. It provides granular, stage-aware visibility into non-terminal shipments to reduce RTO rates, unlock stuck COGS, and improve Customer Experience (CX).

The Control Tower has four main dashboards:
- Forward Movement Dashboard: Tracks outbound shipments from warehouse to customer.
- Reverse Movement Dashboard: Tracks return shipments from customer back to warehouse.
- RTO Movement Dashboard: Tracks failed deliveries being returned to origin.
- Quick Commerce Dashboard: Tracks hyper-local, time-sensitive deliveries.

Navigation path: Home > Control Tower > [Forward / Reverse / RTO / Quick Commerce / Smart Alerts / Configuration]

The Control Tower has a three-tier page structure:
- Landing Page: High-level KPIs and charts for quick scanning.
- Intermediate Page: Breakdowns by courier, warehouse, region, or delivery attempt for diagnosis.
- Dump Page: Granular AWB-level data for direct action and export. Reports can be downloaded and sent directly to your email. Data can be visualized as Shipment Count, Invoice Value, or Quantity using the toggle on the dump page.

Data refreshes every 15 minutes. The system supports 20 concurrent users per brand and retains data for 90 days.

User personas:
- Supply Chain Executive (MIS/Movement Coordinator): Operates daily dashboards to identify delays and coordinate with carriers.
- Supply Chain Manager (Regional/Department Head): Monitors SLA adherence and escalates with carriers.
- Senior Management (CXO/CFO/COO): Reviews high-level metrics, bottlenecks, and long-term trends.

SECTION 6: SMART ALERTS — HOW TO SET UP
================================================================================

Smart Alerts V2 is the automated notification engine for the Control Tower. It sends alerts via Email, WhatsApp, and Slack with attached CSV or XLS shipment data dumps.

Two alert modes:
- Instant Alert: Event-driven — fires when a shipment enters a bucket or a threshold is exceeded.
- Scheduled Alert: Time-driven — fires at a configured time (hourly, daily, or custom).

Smart Alerts are only configurable on the default dashboard. Non-default dashboards cannot have alerts.

--- HOW TO SET UP A SMART ALERT (5 Steps) ---

IMPORTANT: Smart Alerts can only be configured on the default dashboard. Before starting, ensure you are working on the default dashboard. Navigate to: Home > Control Tower > Smart Alerts > Create New Alert.

Step 1 — Select Context (MANDATORY fields):
- Select the Control Tower type: B2C or Quick Commerce.
- The default dashboard is auto-populated and cannot be changed.
- Select the widget/metric you want to alert on. Examples of widgets you can alert on: Pending Pickups, Stuck at Destination Hub, Failed Deliveries, Breached PDD, Delayed In-Transit, RTO Marked Shipments.
- Select the cutoff bucket(s) configured for that widget (e.g., 48+ hours for Pending Pickups, or 36+ hours for Stuck at Destination Hub).
- Select one or more carriers — MANDATORY. You cannot proceed to Step 2 without selecting at least one carrier.

Step 2 — Select Alert Mode (MANDATORY):
- Choose ONE mode only — you cannot select both simultaneously.
- Instant Alert: Event-driven — fires immediately when a shipment enters the bucket or threshold is exceeded.
- Scheduled Alert: Time-driven — fires at a fixed time. Set timing to: Hourly, Daily, Alternate Days, Weekdays only (Monday to Friday), or a Custom time (e.g., every day at 9 AM).

Step 3 — Define Trigger Logic (optional refinements):
- Configure when the alert fires:
  - When a shipment enters the selected bucket.
  - When a shipment remains in the bucket for X minutes.
  - When bucket count or percentage exceeds a defined threshold.
- Optional priority filters to narrow the alert scope:
  - Order Value (e.g., only alert for high-value orders)
  - Region / City / Pincode
  - Store / Warehouse
  - Premium CX
- Note: SLA breach cannot be used as a filter here — the widgets themselves already represent breached or breaching states.

Step 4 — Payload Configuration (MANDATORY — select at least one field):
- Select which data fields appear in the attached file sent with the alert.
- Available field groups to choose from:
  - Order details
  - Shipment stage and ageing
  - Store / hub / city
  - Rider / carrier
  - Invoice value
- Choose attachment format: CSV or XLS.

Step 5 — Channels, Recipients & Escalation (MANDATORY — select at least one channel):
- Select delivery channel(s): Email, WhatsApp, or Slack.
- For Email: provide email IDs.
- For WhatsApp: provide phone numbers.
- For Slack: select available Slack channels.
- Carrier-wise POC details auto-populate based on the carrier(s) selected in Step 1. You can edit or add additional recipients.
- Enable escalation (optional):
  - Set one or more escalation POCs.
  - Set escalation time with units: minutes / hours / days.
  - On escalation, the system re-evaluates the current shipment state and sends a fresh payload of still-unresolved shipments — not the original alert. Escalation fires even if shipments moved to a higher severity bucket since the original alert.
- Click Save to activate the alert.

--- WORKED EXAMPLE: Daily Email Alert for Pending Pickups ---

Goal: Receive a daily email every morning listing all shipments pending pickup for 48+ hours.

Step 1 — Select Context:
- Control Tower type: B2C
- Widget: Pending Pickups
- Cutoff bucket: 48+ hours
- Carrier: Select all relevant carriers (e.g., Bluedart, Delhivery, XpressBees)

Step 2 — Select Alert Mode:
- Mode: Scheduled Alert
- Timing: Daily
- Time: 9:00 AM (or your preferred morning time)

Step 3 — Trigger Logic:
- Trigger: When bucket count exceeds 0 (i.e., any shipment in 48+ hour bucket)
- Priority filters: Optional — add Warehouse filter if you want alerts specific to one fulfillment center

Step 4 — Payload:
- Select fields: Order details, Shipment stage and ageing, Store/hub/city, Rider/carrier
- Format: CSV

Step 5 — Channel and Recipients:
- Channel: Email
- Recipients: Add your ops team email IDs
- Escalation: Optional — add manager email with 2-hour escalation if unresolved

Result: Every morning at 9 AM, you receive a CSV email listing all shipments pending pickup for 48+ hours with full order details.

SECTION 10: NDR → RTO ESCALATION LOGIC
================================================================================

Failed Delivery → RTO escalation process:
- A shipment in the Forward Failed Deliveries widget is still in the forward pipeline — the courier can still attempt reattempt.
- The 3+ days bucket in Forward Failed Deliveries signals critical aging — immediate action needed to prevent RTO.
- If no reattempt is made and tracking goes silent for 48–96 hours, the shipment appears in RTO Potential Shipments widget on the RTO Movement dashboard.
- Once a courier officially marks the shipment for return, it moves to RTO Marked Shipments widget — categorized by how many OFD attempts were made before marking.
- If the RTO shipment takes 7+ days to return, it appears in RTO Stuck in Transit widget.

To monitor a specific failed shipment:
1. Navigate to Forward Movement dashboard.
2. Check the Forward Failed Deliveries widget — find the aging bucket the shipment is in.
3. Drill down to Dump Page to locate the specific AWB.
4. If shipment has been silent for 48+ hours, also check the RTO Potential Shipments widget on the RTO Movement dashboard.

SECTION 14: USER ANALYTICS (POSTHOG)
================================================================================

The Control Tower uses PostHog for user analytics. The dashboard is identified by URL containing /control-tower.

Key metrics tracked:

Daily Active Users (DAU): Unique users performing any event on the Control Tower URL each day.

Dashboard Stickiness: DAU/MAU ratio — measures if the tool is becoming a daily habit.

Sessions per User: Total Sessions divided by Unique Users.

User Lifecycle (Last 30 Days):
- New Users: First-time users of the dashboard.
- Returning Users: Active recently and came back.
- Resurrecting Users: Were inactive, then returned.
- Dormant Users: Were active but have been completely inactive for 14+ days.

Interaction Rate: % of visitors who clicked on a widget or filter (not just viewed the page).
Bounce Rate: % of visitors who viewed only one page and left without clicking anything.

Widget Clicks tracked: Forward Widget Clicks, Reverse Widget Clicks, RTO Widget Clicks, QC Widget Clicks — each broken down by specific widget name.

Time Spent on Dashboard: Grouped into buckets: <1 Min, 1-3 Mins, 3-5 Mins, >5 Mins.

Enterprise-Level Adoption: Daily Users by Enterprise — stacked bar chart by enterprise name.

New Control Tower Users (Last 30 Days): Count of users whose first visit to /control-tower was within the last 30 days.
