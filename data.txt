“B2C Control Tower PRD
1. Context & Problem Statement
In the dynamic e-commerce and D2C landscape, brands grapple with invisible bottlenecks across the shipment journey—first-mile (order to dispatch/pickup), mid-mile (dispatch to out-for-delivery/transit), and last-mile (out-for-delivery to delivered)—that cascade into severe downstream issues. Without granular visibility into where shipments stall (e.g., aging pickups at warehouses, hub processing delays, or failed delivery reattempts), brands suffer from skyrocketing Return-to-Origin (RTO) rates (often 20-30% in COD-heavy ops), degraded customer experience (CX) through unaddressed delays and complaints, and stuck cost of goods sold (COGS) tied up in unresolved inventory. Multi-carrier ecosystems exacerbate this: ops teams can't pinpoint which carriers underperform at specific stages (e.g., Ekart's 71% share in 3+ day transit delays), quantify stuck volumes (e.g., 100K shipments >24 hrs at source hubs), or track stagnation duration (e.g., 36+ hrs since last scan). Manual chases via WhatsApp, Excel, or siloed portals delay resolutions by 4+ hours, while the absence of automated carrier alerts perpetuates reactive firefighting.
Current challenges include:
* Opaque Journey Breakdowns: No real-time mapping of issues to first/mid/last-mile, leading to misallocated resources (e.g., over-focusing on last-mile while mid-mile hubs accounts for 50% of delays).
* Carrier Accountability Gaps: Difficulty identifying stage-specific culprits (e.g., Blue Dart's high first-mile failures) or escalating via automated alerts, resulting in unoptimized SLAs and vendor negotiations.
* Quantification Deficits: Stuck shipment volumes and durations remain buried, inflating RTOs (e.g., 20K without attempts) and locking COGS (e.g., aged exceptions >20 days).
* CX & Cost Cascades: Unflagged delays erode trust (e.g., 420K SLA breaches), spike support tickets, and trap capital in limbo inventory without proactive interventions.
* Scalability Hurdles: Fragmented tools hinder geography/segment analysis, preventing data-driven tweaks like zone-specific thresholds.

Few use cases:
* First-Mile Optimisation: Validate COD orders pre-pickup to cut fake/duplicates, reducing RTO by 40%.
* Mid-Mile Diagnostics: Alert on hub/transit stalls (e.g., >36 hrs) to reroute, freeing COGS faster.
* Last-Mile Recovery: Track failed reattempts by carrier/stage, automating alerts for 95% success.
* Cross-Carrier Escalation: Pinpoint issues (e.g., DHL's 40% delay share) and notify via emails to the carrier POC.


There is a need for a real-time, unified Control Tower that dissects the shipment journey into actionable stages, surfaces carrier/stage-specific pain points, quantifies stuck volumes/durations, and automates carrier alerts—transforming reactive ops into proactive command centers that slash RTOs, unlock COGS, and elevate CX.
2. Objective
Enable brands to gain granular, stage-aware visibility into non-terminal shipments across Forward, Reverse, and RTO movements, empowering ops/CX teams to:
* Dissect Journey Issues: Pinpoint first/mid/last-mile bottlenecks, stuck volumes (e.g., 100K >24 hrs), and durations (e.g., 36+ hrs since scan) for targeted fixes.
* Carrier Performance Tracking: Identify stage-specific underperformers (e.g., Ekart mid-mile delays) with breakdowns by carrier/region.
* Automated Escalations: Trigger real-time/scheduled alerts to carriers/teams on breaches, reducing resolution from 4 hrs to <30 mins.
* Outcome Optimization: Drive RTO reductions (50%), COGS release (via backlog clearance), and CX uplift (proactive comms on risks).
* Data-Driven Iteration: Provide customizable analytics for geography/segment tweaks, boosting SLA adherence (72%→90%).

3. Solution Overview
The B2C Control Tower is a dedicated real time monitoring product within the ClickPost dashboard, engineered for real-time exception management of non-terminal shipments only. It unifies Forward (outbound), Reverse (returns), and RTO (failed returns) into modular dashboards, each with a three-tier flow: Landing (KPIs/charts for scanning), Intermediate (breakdowns for diagnosis), and Dump (AWB traces for action). Data normalises multi-carrier feeds with stage-aware formulas (first/mid/last-mile mappings).
Key features:
* Stage-Aware Metrics: Journey breakdowns (e.g., first-mile: pending pickups; mid-mile: hub/transit delays; last-mile: failed reattempts) with stuck counts/durations.
* Carrier/Granularity Filters: Global slicers (date [15/30-day defaults from created date/ RTO mark date], carrier, region, type, warehouse) for pinpointing issues.
* Automated Alerts: Configurable triggers (e.g., >24-hr mid-mile stall) with carrier-direct emails with Shipment Details Dump
* Visual & Export Tools: Donuts/bars for buckets, sortable tables, CSV/PDF exports; 15-min refreshes.
* Configs: Custom Cutoff time configuration on which brands want to optimise their shipment journey in each miles under each widgets..

7. Technical Requirement
7.1 Data Ingestion:

System supported: CRM, OMS, WMS, Carrier System


1. Control Tower runs on a separate database from the production database.
2. This CT database is used only for Control Tower dashboards and queries.
3. Data is pushed from the production database to CT through Kafka events.
4. Only the required tables (replicated tables) needed for CT queries are maintained in the CT database.
5. When an event is received from Kafka, the CT service updates the corresponding CT tables.
6. Dashboards and queries run on these CT tables so that complex or heavy aggregations do not hit the production DB, avoiding locks and slowness.

7. Technical Requirement
7.1 Data Ingestion:
System supported: CRM, OMS, WMS, Carrier System
                                                                     1. Control Tower runs on a separate database from the production database.
                                                                     2. This CT database is used only for Control Tower dashboards and queries.
                                                                     3. Data is pushed from the production database to CT through Kafka events.
                                                                     4. Only the required tables (replicated tables) needed for CT queries are maintained in the CT database.
                                                                     5. When an event is received from Kafka, the CT service updates the corresponding CT tables.
                                                                     6. Dashboards and queries run on these CT tables so that complex or heavy aggregations do not hit the production DB, avoiding locks and slowness.

                                                                                                                           * The users should be able to rearrange the dashboard tab.

Purpose:
 Measure the performance of appointment-based pickups and deliveries.
Landing Page Metrics:

                                                                                                                                 * Appointment Adherence Rate (%)
                                                                                                                                 * Appointments Breached

Logics:
                                                                                                                                 1. Purpose
Determine whether a shipment will meet its appointment window (marketplace / consignee slot) using:
                                                                                                                                 * Carrier EDD or Committed SLA (enterprise × courier configurable),
                                                                                                                                 * Latest tracking scan bucket (Pickup / InTransit / OFD / Delivered / Failed / Blank),
                                                                                                                                 * Appointment timing rules (buffer + OFD cut-off).
Output is appointment_status mapped to enums:
                                                                                                                                 * BLANK (1), ON_TIME (2), AT_RISK (3), BREACHED (4), SUCCESS (5), EDD_MISSING (6), SLA_MISSING (7).
                                                                                                                                 2. Variables & Config
Shipment / time variables
                                                                                                                                 * T = Appointment End Time (drop end time)
                                                                                                                                 * BT = Buffer Time (hrs)
                                                                                                                                 * configurable at enterprise level
                                                                                                                                 * default 2 hours
                                                                                                                                 * CT = Cut-off time (time-of-day)
                                                                                                                                 * configurable at enterprise level
                                                                                                                                 * default 11:00 AM
                                                                                                                                 * represents the latest time by which shipment should reach OFD on appointment day
                                                                                                                                 * X = Latest scan timestamp (used for OFD logic)
                                                                                                                                 * P = Picked-up scan timestamp (if present)
                                                                                                                                 * A = AWB Registered timestamp
                                                                                                                                 * EDD = Expected Date of Delivery from carrier (date/datetime; use what carrier provides)
                                                                                                                                 * S = Committed SLA (duration; hours/days)
Derived
                                                                                                                                 * T_threshold = T - BT
(latest “safe” time to still be considered on track for the appointment)

Primary source: order_services_orderappointmentdetails
Fields used:
                                                                                                                                    * awb_number
                                                                                                                                    * drop_start_time, drop_end_time (T = drop_end_time)
                                                                                                                                    * courier_drop_start_time, courier_drop_end_time (optional carrier slot; can be used for debugging)
                                                                                                                                    * courier_partner_id, enterprise_user_profile_id, tracking_info_id
                                                                                                                                    * appointment_status (computed + stored)
                                                                                                                                    * api_error_message, api_status (for troubleshooting)
Tracking scan / status bucket inputs (from tracking tables via tracking_info_id):
                                                                                                                                    * Latest scan timestamp X
                                                                                                                                    * Pickup timestamp P (if available)
                                                                                                                                    * Bucket classification for the latest status:
                                                                                                                                    * Pickup, InTransit, OFD, Failed, Blank, Delivered
                                                                                                                                    4. Bucketing Rule (Prerequisite)
All ClickPost status codes are mapped to one of these buckets (as per the referenced mapping sheet).
The widget logic runs on the bucket, not on individual raw status codes.
Bucket values:
                                                                                                                                    * Pickup
                                                                                                                                    * InTransit
                                                                                                                                    * OFD
                                                                                                                                    * Failed
                                                                                                                                    * Blank
                                                                                                                                    * Delivered
If bucket cannot be determined → treat as Blank.
                                                                                                                                    5. SLA Basis Selection (Config)
At enterprise × courier level, user chooses SLA basis:
                                                                                                                                    * Mode = COMMITTED_SLA
                                                                                                                                    * Mode = CARRIER_EDD
This mode applies only to Pickup and InTransit buckets (pre-OFD).
                                                                                                                                    6. Status Computation — High-Level Flow
Step 0: Validate appointment window
If T (drop_end_time) is NULL:
                                                                                                                                    * set appointment_status = BLANK (cannot evaluate)
Else compute:
                                                                                                                                    * T_threshold = T - BT
Step 1: Route by bucket
A) Bucket = Pickup or InTransit
A1) If Mode = COMMITTED_SLA
If S is NULL:
                                                                                                                                    * appointment_status = SLA_MISSING
Else:
                                                                                                                                    * If P is NOT NULL:
                                                                                                                                    * If (P + S) <= T_threshold → ON_TIME
                                                                                                                                    * Else → AT_RISK
                                                                                                                                    * If P is NULL:
                                                                                                                                    * If (A + S) <= T_threshold → ON_TIME
                                                                                                                                    * Else → AT_RISK
Notes:
                                                                                                                                    * S is treated as a duration added to the chosen anchor timestamp.
                                                                                                                                    * Use P as primary anchor; fallback to A when pickup scan not available.
A2) If Mode = CARRIER_EDD
If EDD is NULL:
                                                                                                                                    * appointment_status = EDD_MISSING
Else:
                                                                                                                                    * If EDD <= T_threshold → ON_TIME
                                                                                                                                    * Else → AT_RISK
Note: If EDD is a date-only, interpret it as end-of-day (23:59:59) unless carrier provides a timestamp. (Document this behaviour in PRD as “Assumption: EDD granularity”.)
________________


B) Bucket = OFD
Use latest scan timestamp X (the OFD scan).
If X is NULL:
                                                                                                                                    * appointment_status = AT_RISK
(bucket says OFD but timestamp missing; treat as risk)
Else apply:
                                                                                                                                       * If Date(X) < Date(T) → ON_TIME
                                                                                                                                       * If Date(X) = Date(T):
                                                                                                                                       * If Time(X) <= CT → ON_TIME
                                                                                                                                       * Else → AT_RISK
                                                                                                                                       * If Date(X) > Date(T) → BREACHED
Interpretation:
                                                                                                                                       * OFD prior to appointment day = safe.
                                                                                                                                       * On appointment day, OFD must happen before the configured cut-off time (default 11 AM).
                                                                                                                                       * OFD after the appointment day (Carrier drop time) is breached.
C) Bucket = Delivered
                                                                                                                                       * appointment_status = SUCCESS
Optional (if you want to be stricter later): you can compare delivered timestamp to T to distinguish “delivered within appt” vs “delivered late”, but your current rule sets all Delivered as Success.
D) Bucket = Failed
                                                                                                                                       * appointment_status = BREACHED
E) Bucket = Blank
                                                                                                                                       * appointment_status = BLANK
User Stories:
                                                                                                                                       * As a warehouse manager, I want to identify all shipments where appointments were missed to analyse the root cause.
                                                                                                                                       * As a carrier manager, I want to compare adherence across carriers to improve pickup scheduling.
                                                                                                                                       * As a brand, I want to proactively identify “At Risk” appointments to reallocate resources and avoid breach.

8. HogQL Queries 
When the standard UI isn't enough, HogQL (PostHog's SQL dialect based on ClickHouse) is required. Here are the core queries for your specific metrics, written to respect dashboard filters.
A. Time Spent on Dashboard (The "Buckets" Bar Chart)
To create the bar chart showing how many people stay for different lengths of time, you need to use multiIf to create the buckets.
SQL
SELECT
    multiIf(
        session.duration < 60, '< 1 Min',
        session.duration >= 60 AND session.duration < 180, '1-3 Mins',
        session.duration >= 180 AND session.duration < 300, '3-5 Mins',
        '> 5 Mins'
    ) AS time_spent_bucket,
    count(DISTINCT http://session.id ) AS total_sessions
FROM events
WHERE event = '$pageview' 
  AND properties.$current_url LIKE '%/control-tower%'
  -- The following line ensures dashboard-level date and property filters apply to this query
  AND {filters} 
GROUP BY time_spent_bucket
ORDER BY total_sessions DESC
B. New Control Tower Users (Filtered by Timeframe)
To find out how many brand new users tried the Control Tower, while allowing the dashboard's date filter to change the results.
SQL
SELECT count(DISTINCT distinct_id) AS new_control_tower_users
FROM (
    SELECT 
        distinct_id, 
        min(timestamp) AS first_visit_time
    FROM events
    WHERE properties.$current_url LIKE '%/control-tower%'
    GROUP BY distinct_id
)
-- By using {filters}, if a user selects "Last 7 Days" on the dashboard, 
-- this query automatically updates to only show users whose first visit was in that window.
WHERE first_visit_time >= now() - interval 30 day 
AND {filters}
C. Active Users by Enterprise (Stacked Bar Chart Logic)
While this can be done in the standard UI, if you need to calculate a complex ratio (like active users divided by total paid seats per enterprise), you'd use HogQL.
SQL
SELECT 
    person.properties.enterprise_name AS enterprise,
    count(DISTINCT person_id) AS active_users
FROM events
WHERE event = '$pageview'
  AND properties.$current_url LIKE '%/control-tower%'
  AND {filters}
GROUP BY enterprise
ORDER BY active_users DESC
________________


A quick tip on {filters}: Whenever you write custom HogQL in an insight, always include AND {filters} in your WHERE clause. If you don't, the chart will break or ignore the drop-downs when someone tries to filter the master dashboard by Date or Enterprise!
9. Definitions
Active users - The count of unique visitors who interact with your site within a specific daily, weekly, or monthly window. Formula: Sum of unique user IDs with at least one session in period.
Session duration - The average length of time users spend on your site during a single visit. Formula: Total duration of all sessions / Total number of sessions.
Sessions - A single period of interaction between a user and your website that ends after inactivity. Formula: Total count of individual visit instances.
Pages per session - The average number of distinct pages a user views before leaving the site. Formula: Total pageviews / Total sessions.
New vs returning users - A metric comparing first-time visitors to those who have visited at least once before. Formula: (Number of new users / Total users) vs (Number of returning users / Total users).
Stickiness - A measure of how often users return to your product within a given month. Formula: Daily Active Users (DAU) / Monthly Active Users (MAU).
Retention rate - The percentage of users who continue to engage with your site over a specific time period. Formula: (Users at end of period - New users during period) / Users at start of period.
Custom events - Specific user-defined actions tracked on the site, such as button clicks or video plays. Formula: Total occurrences of a specific tagged action..
Lifecycle Toggles
                                                                                                                                          * New - Users who did an event during the interval and were also created during that period (e.g., created an account and sent a message today).
                                                                                                                                          * Returning - Users who were active in the previous interval and remain active in the current interval (e.g., sent a message yesterday and today).
                                                                                                                                          * Resurrecting - Users who were inactive in the previous interval but became active again (e.g., sent no messages for 10 days, but sent one today).
                                                                                                                                          * Dormant - Users who are inactive in the current interval but were active in the previous one (e.g., sent a message yesterday, but not today).
Retention & Engagement
                                                                                                                                          * Retention - Creates a cohort of unique users who performed an event for the first time in a given week, tracking the percentage who return to perform any event in subsequent weeks.
                                                                                                                                          * Interaction Rate - Out of everyone who opened the page, how many clicked at least one thing? (e.g., 93% of users are active).
                                                                                                                                          * Bounce Rate - Out of everyone who opened the page, how many left after looking at just that single page? (e.g., 30% of users are 'one-and-done'—they check the numbers and close the tab without moving elsewhere).

Forward SLA Shipments
What it counts: Shipments performance against committed delivery timelines
Two key metrics calculated:
                                                                                                                                          * Total Delayed: Shipments that exceeded their promised delivery time
                                                                                                                                          * Total On-time: Shipments that met or beat their promised delivery time
How it works:
                                                                                                                                          * Compares actual delivery progress against the committed SLA (Service Level Agreement)
                                                                                                                                          * Uses days as the measurement unit
                                                                                                                                          * For shipments out for delivery: measures from pickup to out-for-delivery time
                                                                                                                                          * For other shipments: measures from pickup to latest update time
Statuses included: Picked Up, In Transit, Delayed, Exception, Contact Customer Care, Out for Delivery, Delivered, Failed Delivery, Lost, Damaged, Expired, Aged, and hub processing statuses
Key conditions:
                                                                                                                                          * Must have a valid committed SLA
                                                                                                                                          * Only counts shipments with defined delivery promises
Purpose: Measures delivery performance against customer expectations and helps identify whether the company is meeting its delivery commitments on time
 
Forward EDD Shipments
What it counts: Shipments performance against courier partner's estimated delivery dates
Three key metrics calculated:
                                                                                                                                          * Total Delayed: Shipments that missed the courier's original delivery estimate
                                                                                                                                          * Total On-time: Shipments that met or beat the courier's original delivery estimate
                                                                                                                                          * Total Delayed Today: Shipments that became delayed specifically today
How it works:
                                                                                                                                          * Compares actual delivery progress against the first estimated delivery date (EDD) provided by the courier partner
                                                                                                                                          * Uses exact timestamps for precision
                                                                                                                                          * For shipments out for delivery: compares out-for-delivery time to EDD
                                                                                                                                          * For other shipments: compares latest update time to EDD
Statuses included: Picked Up, In Transit, Delayed, Exception, Contact Customer Care, Out for Delivery, Delivered, Failed Delivery, Lost, Damaged, Expired, Aged, and hub processing statuses
Key conditions:

” “1. Introduction
1.1 Problem Statement
The current Control Tower system faces significant operational and performance challenges as it attempts to provide real-time visibility into our logistics operations. As the volume of tracking data increases and the need for instantaneous insights grows, our existing architecture has revealed fundamental limitations that hinder scalability, performance, and maintainability.
1.2 Goal
The system must support real-time data ingestion, millisecond-level query response times, fast aggregations, and flexible dashboard creation, while accommodating continuously growing data streams and handling both heavy write and read operations.
1.3 Current Design Issues
The existing Control Tower implementation suffers from several architectural limitations:
                                                                                                                                          * Topic generation: Each new data field requires creating a new Kafka topic, resulting in an unwieldy and difficult-to-maintain topic landscape.
                                                                                                                                          * Inefficient Data Processing: Each topic requires a dedicated consumer that makes separate calls to the main RDS database to fetch the latest data before inserting it into our Postgres database, creating unnecessary network traffic and processing overhead.
                                                                                                                                          * Performance Bottlenecks: As data volume grows, the Postgres database struggles with complex join operations, severely impacting query performance for dashboard rendering.
                                                                                                                                          * Super Table Anti-Pattern: To work around join performance issues, we've implemented a "super table" (controltower_trackinginfo_v2) containing all data for each tracking ID in a single row. This approach:
                                                                                                                                          * Is inflexible and difficult to extend
                                                                                                                                          * Doesn't scale with additional data fields
                                                                                                                                          * Creates schema management challenges
                                                                                                                                          * Has reached its own performance limits, requiring additional caching layers
                                                                                                                                          * Manual Intervention: Adding new data sources or fields requires significant manual configuration and development effort for table management, consumer setup, etc.
                                                                                                                                          * Limited Responsiveness: The system fails to consistently deliver the millisecond-level response times required for a fluid user experience.
1.4 Proposed Solution
Control Tower 2.0 introduces a complete architectural redesign centered around a unified state management approach to address these fundamental issues:
                                                                                                                                          * Consolidated Data Ingestion: Replacing multiple topic streams with a streamlined Kafka-based ingestion layer that standardizes data input formats.
                                                                                                                                          * Unified State Management: Implementing a central state management system that maintains the complete, current state of each tracking entity in memory, eliminating the need for expensive database joins.
                                                                                                                                          * Event-Driven Updates: Adopting an event-sourcing pattern where state changes broadcast updates to downstream consumers, enabling real-time dashboard updates.
                                                                                                                                          * Purpose-Built Data Storage: Creating widget-specific data tables that contain precisely the data needed for each dashboard component, optimized for their specific query patterns.
This redesign will enable:
                                                                                                                                          * Millisecond response times for filtering and data retrieval
                                                                                                                                          * Simplified on-boarding of new data streams
                                                                                                                                          * Flexible, ad-hoc dashboard creation with minimal effort from developer
                                                                                                                                          * Efficient handling of both read and write-heavy workloads
Control Tower 2.0 addresses the fundamental architectural limitations of the current system while providing a foundation for future scalability and feature development.

2. System Architecture
2.1 Architecture Overview
The architecture follows an event-driven approach with a central state management pattern to optimize for both write-heavy ingestion and read-heavy dashboard operations.
The key architectural components include:
                                                                                                                                          1. Data Sources Layer: External systems (eg. Tracking, NDR, etc) that generate events originally and data updates which need to be tracked and visualized. We expect as part of production system design to have a kafka topic associated with each action which it sends the updates when a data is modified.
                                                                                                                                          2. Ingestion Layer: A Kafka-based streaming layer that standardizes and consolidates data from all sources.
                                                                                                                                          3. State Store Management: The core of the system that maintains the unified state of all tracked entities. It processes incoming data, applies updates to the in-memory state store, and ensures consistency.
                                                                                                                                          4. State Update Broadcasting: The component which propagate state changes to downstream consumers in Kafka topic, enabling real-time updates across the system.
                                                                                                                                          5. Widget-Specific Data Storage: Purpose-built database tables for specific widget query patterns, storing only the necessary data for each dashboard component.
This architecture addresses the limitations of the current system by:
                                                                                                                                          * Eliminating the need for complex joins through purpose-built data structures
                                                                                                                                          * Maintaining state in memory for faster access
                                                                                                                                          * Decoupling data ingestion from consumer processing
                                                                                                                                          * Creating optimized data repositories for specific visualization needs
                                                                                                                                          * Establishing clear boundaries between system components
The design prioritizes performance, scalability, and maintainability while providing flexibility for future enhancements and new data sources.

3. Backfill Process
3.1 Purpose
Enable replay of historical data (by enterprise, date range, etc.) when:
                                                                                                                                          * System logic changes
                                                                                                                                          * New fields are introduced
                                                                                                                                          * Corrections/reflows are needed
3.2 Design
                                                                                                                                          * Single Writer Rule: Only the merged topic updates the state store.
                                                                                                                                          * Retention: High retention is kept only on the merged topic (not on source topics).
                                                                                                                                          * Flow:
                                                                                                                                          1. Identify scope (enterprise/date range).
                                                                                                                                          2. Extract relevant records from merged topic.
                                                                                                                                          3. Republish them back to merged topic as backfill events.
                                                                                                                                          4. Consumers process them as normal updates.
3.3 Idempotency
                                                                                                                                          * Backfill events are tagged/marked so future backfill runs skip already republished data.
                                                                                                                                          * Example:
Original: A A A C C C C B B B
                                                                                                                                          * After backfill: A A A C C C C B B B Ab Ab Ab Bb Bb Bb
                                                                                                                                          * 3.4 Benifits
                                                                                                                                          * Simple management (single topic to replay).
                                                                                                                                          * Avoids race conditions.
                                                                                                                                          * Backfill can be rerun safely without duplicates.

Architecture Layers
1. Ingestion Layer
                                                                                                                                          * Consumes data from multiple Kafka topics (simulated via in-memory queues)
                                                                                                                                          * Validates incoming data against schemas
                                                                                                                                          * Routes data to the unified processing stream

                                                                                                                                          * Maintains real-time state in EventStore (in-memory key-value store)
                                                                                                                                          * Merges data from different sources using common keys (e.g., shipment_id)
                                                                                                                                          * Manages referral relationships between different entity types
3. ETL & Processing Layer
                                                                                                                                          * Transforms and enriches data from multiple sources
                                                                                                                                          * Extracts field values using JSONPath expressions
                                                                                                                                          * Validates data against versioned schemas
4. Serving Layer
                                                                                                                                          * Provides widget-based data access through WidgetService
                                                                                                                                          * Publishes processed data to configured widget tables

Problem Statement:

The existing Control Tower system tightly couples source-specific fields with ingestion and processing logic. As a result, adding a new source or supporting additional fields requires substantial engineering effort, manual updates, and testing. This lack of configuration reduces scalability, slows down feature delivery, and increases the risk of regressions.

Overview
The new Control Tower framework introduces a config-driven ingestion layer that enables seamless on-boarding of multiple data sources with minimal development effort.
Instead of writing source-specific code, users can add a new source by:
                                                                                                                                                                     * Specifying the Kafka source topic
                                                                                                                                                                     * Defining the consumer configuration, which describes how fields are extracted from the Kafka payload
                                                                                                                                                                     * Mapping the source topic to its corresponding consumer configuration
Once configured, the source can be plugged into the system without additional code changes, and data ingestion begins automatically.
All ingested data is stored in a centralized Event Store, which maintains the latest state for each event store key. This ensures that downstream consumers always have access to the most up-to-date representation of each entity, regardless of the source from which the data originated.
This architecture significantly improves scalability, extensibility, and maintainability, enabling faster on-boarding of new sources.


Steps to onboard a new source topic:

1) Check ShipmentField.java file and see if we can use any of the existing enums which are not being extracted anywhere else. Each field can only have one source topic (except event store keys).
2) Add the missing shipment fields and their parameters (source topic, event store key type, shipment field type and datatype.
3) Add the SourceConsumerConfig file and mention the extraction field path schema.
4) Add  mapping of SourceConsumerConfig with Source topic in ControlTowerIngestionDataRouterResource.
5) Add entry in event state metadata for the new topic.
6) Add schema for the newly added event state metadata in src/main/resources/schemas/controltower. Keep the package name same as event state metadata (in lower case).
7) To make changes to schema add new versions for schema in the respective package.
8 ) Add entry in jobs_list.yml in clickpost repo with topic name and type of payload (single/bulk).

1. GENERAL OVERVIEW
The ClickPost Control Tower is a real-time monitoring dashboard designed for logistics and operations teams. It provides granular, stage-aware visibility into non-terminal shipments to reduce Return-to-Origin (RTO) rates, unlock stuck Cost of Goods Sold (COGS), and improve Customer Experience (CX).
The Control Tower has three main personas: Supply Chain Executives (who handle daily exception management), Supply Chain Managers (who monitor SLA adherence across carriers), and Senior Management (who track P&L impacts).
The Control Tower has a three-tier navigation system: the Landing Page (high-level KPIs and charts), the Intermediate Page (breakdowns by carrier or region), and the Dump Page (granular AWB-level data for action).

2. FORWARD MOVEMENT (OUTBOUND SHIPPING) WIDGETS
Forward Movement tracks shipments from the warehouse to the customer.
Forward - Order to Dispatch: Measures orders placed but not yet picked up. Includes statuses like AWB Registered, Order Placed, and Pickup Pending.
Forward - Dispatch to OFD: Measures shipments picked up but not yet Out for Delivery.
Forward - OFD to Delivered: Measures shipments currently Out for Delivery and successfully delivered.
Forward - Exceptions: Tracks shipments marked as Lost, Damaged, Expired, or Aged.
Forward - SLA Shipments: Measures delivery performance against the brand's committed SLA timelines (On-time vs Delayed).
Forward - EDD Shipments: Measures delivery performance against the Courier's Estimated Delivery Date (EDD).
Forward - First Delivery Attempt: Measures the success rate of shipments delivered on their very first attempt yesterday.
Forward - Stuck at Destination Hub: Tracks shipments scanned into the final delivery hub but stuck there for 12-24, 24-48, or 48+ hours.
Forward - Stuck at Source Hub: Tracks shipments scanned into the origin hub but not moving forward.
Forward - Stuck in Transit (Pre-Delivery): Tracks shipments with absolutely no tracking updates for 48-72, 72-96, or 96+ hours.
Forward - Breaching PDD: Early warning system for shipments approaching their Promised Delivery Date within 24 hours.
Forward - Breached PDD: Shipments that have already missed their Promised Delivery Date.
Forward - RTO Without Attempt: Shipments that were returned to the origin without a single delivery attempt being made to the customer.
Forward - Pending Pickups: Orders waiting to be picked up from the warehouse, grouped by aging (0-24 hrs, 24-48 hrs, 48+ hrs).

3. REVERSE MOVEMENT (RETURNS) WIDGETS
Reverse Movement tracks shipments being returned by the customer back to the warehouse.
Reverse - Not Picked: Tracks return shipments awaiting initial pickup from the customer.
Reverse - In-Transit: Monitors return shipments currently moving through the logistics network back to the merchant.
Reverse - Pickups SLA: Measures if reverse shipments were picked up from the customer within the 24-hour SLA commitment.
Reverse - Pending Pickups Aging: Groups pending reverse pickups into buckets of 24-36 hours, 36-48 hours, and 48+ hours.
Reverse - Pickup Attempted: Analyzes failed return pickup attempts, separating quality check failures from standard operational failures.
Reverse - Pickup Failed Reason: Groups reverse pickup failures by specific reason codes to identify root causes.

4. RTO MOVEMENT (RETURN TO ORIGIN) WIDGETS
RTO Movement tracks failed deliveries that are being forced back to the origin warehouse.
RTO - Potential Shipments: Active shipments with prolonged tracking silence (48+ hours) that are at high risk of being marked as RTO.
RTO - Marked Shipments: Shipments officially marked for RTO, categorized by how many delivery attempts were made (0, 1, 2, or 3+ attempts) before the return decision.
RTO - Stuck In Transit: RTO shipments taking an unusually long time to return to the warehouse (grouped by 7-10 days, 10-12 days, and 12+ days).
RTO - High Pincodes: Identifies the top 15 delivery destination pincodes that generate the highest volume of RTO shipments.
RTO - Reason: Categorizes RTO shipments by their Non-Delivery Report (NDR) reason codes.

5. B2B CONTROL TOWER FEATURES
B2B Control Tower handles business-to-business bulk shipping.
B2B - Appointment Adherence: Measures if B2B shipments meet their scheduled marketplace/consignee appointment windows. Statuses include On Time, At Risk, Breached, and Success. The calculation uses Buffer Time (default 2 hours) and Cut-off Time (default 11 AM).
B2B - AI Proof of Delivery (POD): Uses Artificial Intelligence to verify POD document accuracy. It compares the AI validation result against manual reviewer overrides to flag correct, missing, or problematic PODs for invoice reconciliation.

6. SMART ALERTS V2
Smart Alerts V2 is the automated notification engine for the Control Tower.
It supports both Instant (event-driven) and Scheduled (time-driven) alerts.
Alerts can be sent via Email, WhatsApp, and Slack with attached CSV/XLS data dumps.
Escalation Logic: If an alert escalates, the system evaluates the shipment's current state. It generates a fresh payload containing only unresolved shipments, even if the shipment has moved to a higher severity bucket since the original alert.

7. TECHNICAL ARCHITECTURE (CONTROL TOWER 2.0)
The Control Tower 2.0 architecture is built for real-time data ingestion and millisecond-level queries.
It moves away from complex PostgreSQL joins and uses an event-driven State Management approach.
Ingestion Layer: Uses a config-driven Kafka system where new data sources map to a central merged topic via "ShipmentFieldConsumerConfigEnum".
State Management: Uses an In-Memory State Store to maintain the latest snapshot of a shipment.
Database: Evaluated Hazelcast, Apache Ignite, and Aerospike. Aerospike is the preferred database because of its sub-millisecond latency, SSD optimization, strong consistency, and excellent handling of high-ingest write volumes.

8. ANALYTICS AND METRICS (POSTHOG)
Control Tower uses PostHog for user analytics.
Daily Active Users (DAU) and Dashboard Stickiness (DAU/MAU ratio) measure engagement.
Time Spent on Dashboard uses a custom HogQL query to group sessions into buckets (<1 Min, 1-3 Mins, 3-5 Mins).
Interaction Rate tracks the percentage of users who actively click widgets versus just viewing the page.

9. CLIENT OPERATIONS AND PRICING
Avimee Herbals Operations: Drashti handles First-Mile (pending dispatches), Goutam handles Mid-Mile (automated CX recovery and SLA monitoring), and Krishna handles Last-Mile (RTO manual calls and financial reconciliation).
Kapiva Focus: Kapiva uses the dashboard primarily to track Same Day Delivery (SDD) and Next Day Delivery (NDD) SLAs, requiring specific Order-to-First-Attempt (O2FA) metrics and Dark Store pincode mapping.
Pricing Model: Enterprise setup fees vary. For example, The Souled Store (₹50,000 with 5% annual increment), Kapiva (₹25,000), Avimee Herbals (₹85,000). Quick Commerce clients like Newme pay ₹25,000 for 5,000 High-Level AWBs, plus ₹1 for each additional AWB.

10. WMS API ENDPOINTS
All APIs route through the base URL: http://0.0.0.0:8000/api/v1/dashboard/control-tower/wms/
APIs require a 'key' parameter and a JSON payload with 'warehouse_id' and 'widget_id'.
/overview/ returns total orders, fulfillment stats, and daily run rate.
/dispatch-readiness/ returns percentage of orders ready for dispatch.
/out-of-stock/ and /top-near-out-of-stock/ track inventory depletion.
/channel-wise-sla-breached/ tracks SLA violations filtered by sales channels like Shopify.

11. DASHBOARD BUILDER AND CUSTOMIZATION
The Control Tower allows users to build dashboards from scratch or use pre-configured templates.
Users can drag and drop widgets, reorder them, and resize the dashboard grid into 2, 3, or 4 columns.
Users can configure custom time cutoff buckets (e.g., 4–8 hrs, 8–16 hrs) that apply globally across all widgets on that specific dashboard.
The system automatically saves layout drafts during editing and prevents duplicate widgets from being added to the same dashboard.

12. GLOBAL FILTERS AND SYSTEM PERFORMANCE
The dashboard includes global filters that apply across all widgets to pinpoint issues. These include Date Range, Carrier Partner, Customer, Warehouse, Region/City/Zone, and Order Type (Prepaid vs. COD).
The Control Tower system targets a Page Load Time of 4 seconds or less and a Filter Apply Response time of 4 seconds or less.
The data pipeline refreshes every 15 minutes using an incremental load based on the 'updated_at' timestamp.
The system is designed to support 20 concurrent users per brand and retains data for a maximum of 90 days.

13. SMART ALERTS V2 SYSTEM INTELLIGENCE
The Smart Alerts system uses intelligence rules to prevent duplicate alerts for the same shipment within a defined time window.
The system will automatically suppress alerts if shipments successfully move out of all applicable delay buckets.
Every alert message includes a summary showing the total shipments, total invoice value, oldest aging shipment, and the number of new versus resolved shipments since the last alert.
Users can apply optional priority filters to Smart Alerts based on Order Value, Region, Store, or Premium CX status.

14. FORWARD MOVEMENT - STUCK FAILED DELIVERIES WIDGET
The Stuck Failed Deliveries widget tracks shipments that failed their first delivery attempt and remain stuck without any subsequent delivery progress.
It tracks the aging of these failed deliveries in three buckets: 0-2 days (recent failures likely in retry queues), 2-3 days (moderate delays needing attention), and 3+ days (critical delays requiring immediate intervention to prevent RTO).
It strictly filters for shipments that have exactly one failed delivery recorded (NDR count = 1) and excludes shipments already marked for return.

15. REVERSE MOVEMENT - AGING METRICS WIDGETS
The Reverse Delivered Aging widget analyzes the complete cycle time from reverse shipment creation to the final return delivery. It groups these transit times into 0-7 days, 7-10 days, and 10+ days.
The Reverse In-Transit Aging widget monitors potential delays for returns that have been picked up but are currently stuck in transit. It groups them by 0-7 days, 7-10 days, and 10+ days since the pickup date.

16. POSTHOG USER LIFECYCLE AND SESSION METRICS
PostHog User Lifecycle metrics categorize the health of the Control Tower user base.
"New Users" are people using the dashboard for the very first time.
"Returning Users" are consistent regulars who used the dashboard recently and returned.
"Resurrecting Users" are past users who stopped using the tool for a while but have returned.
"Dormant Users" are users who were active but have been completely inactive for 14 or more days.
"Single-Page Sessions" (Bounce Rate) measures the percentage of visitors who view only one page and leave without clicking any widgets or filters.

17. CONTROL TOWER 2.0 BACKFILL PROCESS
The Control Tower 2.0 Backfill Process is used to replay historical data when system logic changes, new fields are introduced, or corrections are needed.
It follows a Single Writer Rule where only the merged topic is allowed to update the state store.
Backfill events are tagged to ensure idempotency, meaning the backfill process can be rerun safely without creating duplicate records or race conditions.

18. CONTROL TOWER 2.0 CORE CLASSES AND CONFIGURATION
The Control Tower 2.0 ingestion process is config-driven to allow seamless onboarding of new data sources.
The "ShipmentFieldEnum" defines all trackable fields (like shipment_id or pickup_date) alongside their data types.
The "ShipmentFieldConsumerConfigEnum" specifies the source Kafka topic and the schema mapping used to extract data paths.
The "EventStore" is an in-memory key-value store that maintains the latest event data and tracks referral relationships between different entities (like linking a shipment_id to an ndr_id).
The "MockMergedConsumer" extracts fields from all sources, validates them against versioned schemas, and publishes them to the unified state management system.

19. B2B APPOINTMENT ADHERENCE CALCULATION LOGIC
The B2B Appointment Adherence widget evaluates if a shipment will meet its marketplace or consignee slot using three main variables: T (Appointment End Time), BT (Buffer Time, which defaults to 2 hours), and CT (Cut-off Time, which defaults to 11:00 AM).
The widget calculates a safe threshold time (T_threshold) by subtracting the Buffer Time (BT) from the Appointment End Time (T).
If the appointment window (T) is missing or NULL, the widget automatically sets the appointment status to "BLANK".
For shipments in the "Pickup" or "InTransit" bucket, if the pickup time plus the committed SLA is less than or equal to the safe threshold (T_threshold), the status is "ON_TIME". If it is greater, the status is "AT_RISK".
For shipments in the "Out for Delivery" (OFD) bucket, if the OFD scan happens before the appointment day, the status is "ON_TIME". If it happens on the appointment day but before the 11:00 AM cut-off time (CT), it is "ON_TIME". If it happens after the cut-off time, it is "AT_RISK".
If an Out for Delivery (OFD) scan happens after the appointment day entirely, the B2B widget marks the appointment status as "BREACHED".
If a shipment is in the "Failed" bucket, the appointment status is immediately marked as "BREACHED".

20. B2B AI PROOF OF DELIVERY (POD) STATUS LOGIC
The B2B AI Proof of Delivery (POD) widget determines the final verification status of a delivery receipt by comparing the AI-evaluated status against the manual reviewer override status.
If the manual reviewer status is "PENDING", the final status defaults to whatever the AI evaluated (status_ai).
If the manual reviewer status is "CORRECT" or "PROBLEMATIC", the manual reviewer's decision overrides the AI, and the manual status becomes the final status.
If the manual reviewer status is missing or NULL entirely, the widget treats it as "PENDING" and falls back to using the AI-evaluated status.
If the manual status is "PENDING" but the AI status is also NULL or missing, the final POD status is marked as "PENDING".

21. WMS DASHBOARD API BUCKET INDEX LOGIC
The WMS Control Tower APIs use specific "bucket_index" integers to filter data ranges for dashboard widgets.
For the WMS Out of Stock API (widget_id 41), bucket_index 0 equals 0-1 days, bucket_index 1 equals 2-4 days, bucket_index 2 equals 5+ days, and bucket_index 3 equals "Never Ordered".
For the WMS Dispatch Readiness API (widget_id 40), bucket_index 1 represents 0-30 percent readiness, bucket_index 2 represents 30-60 percent, and bucket_index 3 represents 60-100 percent.
For the WMS Top Near Out of Stock API (widget_id 44), bucket_index 1 means stock will run out in 0-2 days, bucket_index 2 means 2-5 days, and bucket_index 3 means 5+ days.
For the WMS Stock Forecast API (widget_id 42), bucket_index 1 covers 0-7 days, bucket_index 2 covers 8-15 days, and bucket_index 3 covers 15+ days.

22. FORWARD MOVEMENT DETAILED WIDGET LOGIC
The Forward SLA Shipments widget calculates delays by comparing actual delivery progress against the committed Service Level Agreement (SLA). It measures out-for-delivery shipments from the pickup time to the out-for-delivery time.

The Reverse Not Picked widget logic strictly filters for shipments handled by Reverse Logistics Partners (RVP) where the pickup date is officially recorded as NULL in the system.
The Reverse Pickup Attempted widget tracks failed attempts by checking that no pickup has been completed, the Non-Pickup Report (NPR) count is between 1 and 3, and the order is not in a CANCELLED_ORDER status.

24. RTO MOVEMENT DETAILED WIDGET LOGIC
The RTO Without Attempt widget logic identifies shipments where the RTO process has started (using the RTO mark date) but explicitly verifies that the Out For Delivery (OFD) count is exactly zero.
The RTO Marked Shipments widget analyzes the relationship between delivery effort and returns by grouping officially marked RTO shipments by their Out For Delivery (OFD) attempt count (0, 1, 2, or 3+ attempts) prior to the return decision.

RTO Marked Shipments
What it counts: Shipments that have been officially marked for Return to Origin (RTO), categorized by the number of delivery attempts made before marking

Delivery attempt buckets:

0 attempts: Shipments marked for RTO without any delivery attempts

1 attempt: Shipments marked for RTO after 1 delivery attempt

2 attempts: Shipments marked for RTO after 2 delivery attempts

3+ attempts: Shipments marked for RTO after 3 or more delivery attempts

How it works:

Counts shipments that have been officially marked for return process

Groups them by the number of out-for-delivery attempts made before RTO marking

Tracks delivery effort invested before deciding to return shipment

Uses RTO mark date to identify when return decision was made

Statuses included:

RTO_REQUESTED (11)

RTO (12)

Key conditions:

Must have RTO mark date (officially marked for return)

Currently in RTO requested or initiated status

Excludes reverse pickup services

Tracks delivery attempts via OFD (Out For Delivery) count

Purpose: Analyzes the relationship between delivery attempts and RTO decisions, helping identify if shipments are being returned too quickly (0-1 attempts) or if delivery teams are making sufficient effort before marking returns. Critical for optimizing delivery attempt policies and reducing unnecessary RTO costs while maintaining operational efficiency.
